# 把新闻抓取做成一个自动化编辑部：我如何用 LangGraph.js 搭了一套日报系统

做信息消费的人，大概率都经历过同一种疲劳：  
RSS 订了一堆，Folo 里也加了很多源，热榜每天也在刷，但真到晚上回头看，脑子里只剩下模糊的“今天信息很多”，却没有一份可以沉淀判断的结果。

所以我最后没有继续做“更好看的阅读器”，而是做了另一种东西：**一套自动化编辑部**。

这个项目叫 `rss-pub-agent`。它的目标不是展示所有新闻，而是把每天过去 24 小时的海量资讯，压缩成一份有结构、有优先级、有判断的日报，同时再派生出播客脚本、平台分发文案和站点展示页。

换句话说，它解决的不是“怎么抓更多”，而是“怎么把信息变成判断”。

## 一、我想做的不是 RSS 阅读器，而是一条编辑流水线

如果只是做聚合，工程上并不复杂：抓 feed、去重、渲染列表，差不多就结束了。  
但真正麻烦的是下面这些问题：

1. 每天原始输入太多，人工不可能稳定扫完。
2. 很多信息只是重复转载，或者只有热度没有判断价值。
3. 同一天里不同赛道的信息密度差异很大，AI 很容易淹没掉政策、市场和软件工程。
4. 如果日报、周报、播客、平台摘要分别维护，会迅速演变成一堆孤岛。

所以我最后把问题定义成了一条流水线：

```text
抓取 -> 事件压缩 -> 编务判断 -> 去噪 -> 评分 -> 深度解读 -> 多格式输出
```

只要这条链路足够稳定，前端、播客、平台分发都只是“消费同一份结果”的不同壳子。

## 二、架构核心：14 个节点，一条主路径，两个并行输出分支

整个系统用 LangGraph.js 编排，主流程长这样：

```text
START
  -> loadConfig
  -> fetchPrimary
  -> preFilter
  -> fetchCoverage
  -> buildEditorialAgenda
  -> gateKeep
  -> score
  -> enrichSelected
  -> insight
  -> generateDaily
  -> podcastGen ----\
  -> platformsGen --+-> publish
  -> notify
  -> END
```

这里有三个关键决策。

### 1. 先做“编务判断”，再做筛选和评分

很多自动摘要系统一上来就是“抓了就打分”。  
但我发现这样会有一个问题：模型会在没有上下文的情况下逐条判断，结果很容易变成“局部最优”，而不是“今天到底该讲什么”。

所以我在 `gateKeep` 和 `score` 之前插了一个 `buildEditorialAgenda` 节点，让模型先回答几个更像编辑部的问题：

- 今天的 dominant narrative 是什么？
- 哪些主题必须覆盖？
- 哪些分类今天应该被临时提高权重？
- 哪些信号值得继续盯？

这样后续节点就不是在真空里评分，而是在“当天语境”里做决策。

### 2. 输入不做一锅炖，而是分成主抓取和补抓取

项目现在有 28 个 feed source，分成 7 个分类。  
但系统不会直接把 28 个源完全抓平，而是先走一层主输入，再按缺口补抓。

第一层是 `fetchPrimary`：只抓主 `folo-list`，形成当天的核心输入快照。  
第二层是 `fetchCoverage`：如果某些分类明显不够，再定向抓取对应的主池 feed。

这样做的好处很直接：

- 主输入更稳定，不会因为某个外部源波动而导致整体输入形态漂移。
- 覆盖更可控，不会被单一热门赛道占满。

### 3. 输出层并行，但发布点统一

日报正文生成后，播客脚本和平台文案分别并行跑。  
但真正的落盘只有一个 `publish` 节点。

这样能同时满足两个目标：

- 缩短整体耗时
- 保持最终产物的一致性和目录结构稳定

## 三、这个系统最重要的，不是抓新闻，而是怎么决定什么值得写

如果只看工程表面，这像是一个“RSS + LLM”的项目。  
但真正决定质量的，其实是中间那层编辑决策逻辑。

我现在把它拆成三层：

### 第一层：Gate-Keep

模型不是简单输出“保留/删除”，而是做结构化决策：

- `PASS`
- `DROP`
- `MERGE`

这能把广告、水文、重复快讯和标题党尽量压在前面，不让它们进入更贵的后续节点。

### 第二层：六维评分

每条候选会按六个维度评分：

- `signalStrength`
- `futureImpact`
- `personalRelevance`
- `decisionUsefulness`
- `credibility`
- `timeliness`

但这还不够。  
如果只按总分排，日报很容易被单一赛道“刷榜”。

所以我又叠加了三层修正：

- 分类基础权重
- 当天 `editorialAgenda` 的 category boost
- 最低分类覆盖保障

结果就不是“今天最热的 20 条”，而是“今天最值得形成判断的 20 条”。

### 第三层：Insight

进入 `insight` 节点后，系统才开始生成真正可读的内容：

- 一句话总结
- 事件描述
- 为什么值得关注
- 更完整的解读
- 可选的图、代码片段和对比表格

到这一步，新闻才真正从“原始信号”变成“编辑结果”。

## 四、为什么我把内容落成文件，而不是直接塞数据库

这个项目目前的主契约不是数据库，而是 `content/` 目录。

每天会生成：

```text
content/YYYY-MM-DD/
├── daily.md
├── meta.json
├── brief.md
├── douyin.md
├── xhs.md
├── podcast-script.md
└── raw/
```

我这么做有两个考虑。

第一，**内容天然适合版本化**。  
日报、脚本、摘要本质上都是文档，直接用 Git 管理非常自然。

第二，**生产和展示可以彻底解耦**。  
前端只关心 `content/` 是否满足契约，不关心上游是 LangGraph、Cron，还是别的什么东西。

这也是为什么我后面做前端时，整个 `frontend/` 几乎只是在读取文件，而不是参与内容生成。

## 五、前端其实很“克制”：它不生产内容，只消费结果

当前展示层是 Next.js 16。  
它做的事情非常明确：

- 首页读取 `content/index.json`
- 详情页读取 `content/YYYY-MM-DD/daily.md`
- 周报不是独立文件，而是运行时基于日报动态聚合
- 播客页读取 `podcast-script.md`

这种分工我很喜欢，因为它让系统边界非常清晰：

- `scripts/` 负责生产
- `frontend/` 负责展示
- `content/` 是契约

只要这条边界不破，前端可以重构，pipeline 也可以重构，彼此都不会互相拖累。

## 六、一个 LLM 系统真正难的地方，是不稳定时怎么办

如果只是 happy path，这个项目没什么特别。  
真正难的是：某个 provider 超时了怎么办？Folo 没返回完整内容怎么办？跑到第 8 个节点失败了怎么办？

我现在用了几层保险：

### 1. 多 provider 链式回退

LLM 层统一封装，支持 `zhipu / gemini / openai / deepseek / siliconflow`。  
只要配置了 key，就可以进入优先级链。

遇到 `429 / 503 / overloaded / network` 时会自动重试或跳过冷却中的 provider。

### 2. Raw 快照恢复

`fetchPrimary`、`preFilter`、`fetchCoverage`、`editorialAgenda`、`enrichSelected` 等关键中间结果都会落到 `content/<date>/raw/`。

这样一来，如果链路在中途失败，我不需要重新抓取、重新打分、重新跑整条链，只要用 `--resume-from-raw` 从快照继续就行。

### 3. 节点级错误隔离

每个节点都独立 try-catch，把错误写到 `state.errors`。  
只要不是致命问题，后续节点就继续推进，尽量保住当天产物。

这点非常重要，因为对“日报系统”来说，**不完美地发出来**，通常比“因为某个节点失败而整天断更”更有价值。

## 七、这套系统给我最大的提醒：内容系统最终拼的不是模型，而是结构

很多人会把类似项目理解成“模型写日报”。  
但我越做越觉得，模型只是中间的一层能力放大器。

真正决定系统质量的，是这几个更底层的结构性问题：

- 输入源怎么治理
- 怎么做覆盖和优先级平衡
- 怎么把编务逻辑显式化
- 怎么把中间结果落成可恢复的契约
- 怎么让多输出场景复用同一条内容供应链

换句话说，**模型负责增强内容质量，结构负责决定系统是否可持续**。

## 八、如果你也想做类似系统，我建议先想清楚三件事

第一，不要先从“怎么总结新闻”开始，而要先从“你到底想保留什么判断”开始。  
第二，不要把所有数据源一锅端平，先设计主输入和补输入。  
第三，不要急着做花哨前端，先把内容契约和失败恢复做好。

这也是 `rss-pub-agent` 到现在最让我满意的地方：  
它还远没到“终态”，但已经从一个抓资讯的小脚本，变成了一条真正可以每天产出内容的编辑流水线。

如果再用一句话总结这个项目，我会说：

> 我不是在做一个“看新闻”的工具，而是在做一个“把新闻变成判断”的系统。
