# 每日日报中文标题与详情页重构设计

日期：2026-03-29

## 背景

当前每日日报存在三类问题：

1. `daily.md` 中的主新闻标题大量保留英文原文，详情页顶部焦点列表和正文标题不统一。
2. 详情页顶部同时存在“本期焦点”“内容概览”“今日概览”三套近似信息，层级重复。
3. 正文里的评分与来源通过 blockquote 单独展示，视觉上与“事件/解读”割裂，右侧也缺少基于日报正文的目录导航。

这次改造的目标是同时收紧生成侧与渲染侧，让日报 Markdown 自身就具备更好的中文可读性，并让详情页的结构围绕同一份正文数据展开。

## 目标

- 生成出来的 `content/<date>/daily.md` 中，主新闻标题直接写中文。
- 来源继续保留英文原文名称与原始链接。
- 移除正文中的独立“今日概览”表格。
- 将详情页顶部“本期焦点”和“内容概览”合并为一块统一概览区。
- 分类展示统一改为中文标签，不再直接输出 `ai`、`investment` 等英文值。
- 为详情页增加右侧目录，只展示大分类章节和新闻标题。
- 将评分与来源从 blockquote 改成标题右侧紧凑标签，分数保留 100 分制。
- 保证旧日报内容仍可打开，不因新结构上线而失效。

## 非目标

- 不改周报页的整体信息架构。
- 不引入新的结构化持久化文件格式，例如额外生成 `daily.json`。
- 不回写历史 `content/` 中已生成的旧日报文件。
- 不调整现有评分逻辑，只调整展示与输出格式。

## 方案比较

### 方案 A：仅改 Markdown 文案

- 在生成侧把标题翻成中文，删除“今日概览”。
- 前端仍完全依赖 `ReactMarkdown` 直出。

优点：

- 改动最小，风险低。

缺点：

- 标题右侧评分/来源标签很难稳定实现。
- 右侧目录只能做很弱的 heading 抽取，无法和正文卡片样式统一。
- 后续如果继续调详情页排版，会反复受限于原始 Markdown 结构。

### 方案 B：半结构化解析并保留 Markdown 作为源数据

- 生成侧补充中文标题字段并直接写入 `daily.md`。
- 前端先从 Markdown 中解析出章节与新闻条目，再渲染成自定义卡片与目录。

优点：

- 能同时满足中文标题、标题右侧标签、目录和顶部概览合并。
- 仍保留 Markdown 作为单一内容源，便于生成、存档和外部阅读。
- 可对旧内容保留兼容路径。

缺点：

- 需要同时改生成与渲染两条链路。

### 方案 C：改为结构化 JSON 主渲染

- 生成日报时额外写结构化文件，前端完全按 JSON 渲染。

优点：

- 最容易做复杂 UI。

缺点：

- 范围过大，会引入历史兼容、内容双写和维护成本问题。

## 选型

采用方案 B。

理由：

- 用户提出的需求不是纯样式调整，而是内容结构、标题语言、顶部信息层级和目录导航的一次协同重构。
- 方案 B 在控制改动范围的前提下，能够让 `daily.md` 继续作为事实来源，同时赋予详情页足够的结构化渲染能力。

## 设计

### 1. 数据模型与提示词

在 `scripts/lib/types.ts` 的 `NewsInsight` 中新增 `titleZh` 字段，用于保存新闻的中文展示标题。

在 `scripts/nodes/insight.ts` 的 LLM 输出中新增 `titleZh`：

- 约束为中文标题，允许保留必要的英文专有名词。
- 标题长度保持新闻标题风格，不写成摘要句。
- 来源字段继续使用原始 `source` 与 `url`，不做翻译。

这样做的原因是：

- 中文标题需要进入最终 `daily.md`，因此应在生成阶段定型，而不是在前端临时翻译。
- 顶部焦点列表、正文标题、右侧目录都可以复用同一字段，避免多处二次翻译造成不一致。

### 2. 新的日报 Markdown 结构

`scripts/nodes/generate-daily.ts` 将维持“前言 + 分类章节 + 条目内容 + 结尾观察 + 更多 24h 资讯”的总体结构，但调整条目表现形式。

正文主区的新结构如下：

```md
## AI

### OpenYak 发布开源协作桌面客户端

评分 92 · 来源 [Hacker News - wangzhangwu](...)

OpenYak 发布开源协作平台，支持本地运行任意模型并接管文件系统。

**事件：** ...

**解读：** ...
```

具体变化：

- 删除 `## 今日概览` 表格。
- `###` 标题直接使用 `titleZh`。
- 评分和来源改成标题后的一行纯文本元信息，不再使用 blockquote。
- `oneLiner` 改为普通段落，作为标题下的导语，不再作为引用块。
- “更多 24h 资讯”列表保留现状，避免范围扩散。

### 3. 详情页顶部概览合并

`frontend/app/[date]/page.tsx` 里的顶部 hero 区合并为一个统一模块，不再分裂成“本期焦点”和“内容概览”两栏各自命名。

统一概览区包含三部分：

- 焦点标题列表：来自日报正文中的前 3 个 `###` 标题。
- 关键信息指标：发布日期、资讯数、均分、是否含图片、是否含播客。
- 覆盖分类：来自 `meta.categories`，但统一映射为中文标签。

展示原则：

- 页面上不再单独出现第二个“今日概览”标题。
- “覆盖分类”等标签全部改为中文，不直接暴露英文分类 key。
- 页面标题字号略缩小，为标题右侧元信息留出视觉余量。

### 4. 正文解析与自定义渲染

`frontend/components/DailyReport.tsx` 不再只做通用 `ReactMarkdown` 直出，而是改成“解析 + 渲染”的两阶段模式：

1. 解析 Markdown 中的 `##` 与 `###`，识别章节与条目边界。
2. 对主区条目抽取以下结构：
   - 分类章节名
   - 新闻标题
   - 元信息行
   - 导语
   - 事件段
   - 解读段
   - 其他补充块，如图片、对比表、代码片段
3. 用自定义卡片渲染新闻条目。

条目卡片的视觉规则：

- 标题位于左侧。
- 评分和来源以紧凑标签置于标题右侧。
- 导语紧跟标题区，不再使用引用框。
- “事件/解读”维持正文式布局，避免单独的装饰性模块感。

保留 `ReactMarkdown` 的范围：

- 对正文内部的普通段落、图片、表格、代码块继续复用现有 Markdown 组件渲染。
- 仅将顶层章节与新闻条目这层结构交给自定义布局控制。

### 5. 右侧目录

新增一个日报目录组件，数据直接来自正文解析结果，而不是再次扫描 DOM。

目录规则：

- 只展示 `##` 分类章节。
- 每个分类下只展示该章节内的 `###` 新闻标题。
- 不展示“接下来要盯的变量”“更多 24h 资讯”等附加部分。
- 目录在桌面端显示为 sticky rail，移动端隐藏。

布局位置：

- 归档 rail 继续留在左侧。
- 正文保持居中主列。
- 新目录 rail 放在正文右侧，仅在较宽视口显示。

### 6. 分类标签中文化

将分类 key 与中文展示文案的映射收敛到前端公用展示层，而不是在页面里直接输出原始值。

建议把分类显示映射统一放到 `frontend/lib/display-text.ts`，供以下位置复用：

- 日报详情页顶部覆盖分类
- 周报或首页若未来复用该字段
- 右侧目录或其他摘要组件中的分类名

### 7. 旧内容兼容策略

历史 `daily.md` 已经存在 blockquote 风格的评分/来源写法，且标题未必是中文。

兼容策略如下：

- 新生成日报采用新结构。
- 前端解析器优先识别新结构。
- 若遇到旧结构：
  - `###` 标题照常使用原文标题。
  - 若标题后的第一个 blockquote 符合“评分 + 来源”格式，则转换为旧结构元信息。
  - 若导语仍在 blockquote 中，则退化为旧展示。
- 顶部焦点与右侧目录均直接基于 `###` 标题，因此历史页仍可正常渲染，只是不会自动补中文标题。
